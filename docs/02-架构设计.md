## 四、技术架构（架构师视角）

### 4.1 拓扑与数据流（固化现有裁决，补齐契约）

```text
[法国节点] Collector：cron 拉 GSC + DataForSEO
   → 产 artifact（JSON + sha256 + 版本号，契约见 4.2）
   → HTTPS 推送 [广州节点]
[广州节点] 指挥中心：Ingestor 校验入库 SQLite
   → M1-M4 四模块同库消费
   → 飞书日报/告警推运维群（正式启用后）
[Mac mini] Hermes/任务账本：通过 REST 写事件、读进度；任务卡号是跨系统主键
[西班牙 VPS 生产站] 只被 M2 发布中心触碰：审批通过→备份→SSH执行→健康检查→异常自动回滚
```

### 4.2 关键契约（防四机漂移的根）

- **artifact 契约**：`{source, property, date_range, schema_ver, sha256, payload}`——采集消费双方唯一接口
- **统一数据库 schema**：plans（计划树）/ plan_items（分类条目）/ release_plans / change_items / backups / events（统一事件ID：`EV-{date}-{seq}`，字段含 actor/action/input/output/evidence/task_id）/ metrics_daily（GSC+DFS 日指标）
- **Agent API**：`POST /api/events`、`GET /api/plans`、`POST /api/release-plans`，token 鉴权，agent 只能写事件和提案，审批动作只有你能点
- **任务卡号 T-xxx 是贯穿主键**：计划条目、事件、发布单都挂卡号——飞书、账本、指挥中心三边对齐

### 4.3 技术选型（全部复用现有能力，零新技术栈）
Next.js + Prisma + SQLite（团队刚在审批页 P0 验证过此栈）；大屏 = 同应用的 /dashboard 路由全屏模式；访问安全：广州节点仅内网 + 你已接受的法国 Xray 隧道访问，不暴露公网。
